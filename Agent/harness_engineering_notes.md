# 《Harness Engineering》读后笔记

## 第一点：Harness Engineering 的含义

### 传统模式
人 → 给Agent任务 → Agent执行 → 反馈给人 → 人再指导下一步

问题：开发过程总是需要人参与。

### Harness Engineering 模式
人 → 构造环境 → Agent自行判断执行结果是否正确 → 反馈给Agent继续执行 → 直到成功

形成**闭环**，人不用长时间参与。

### 具体例子
前端开发任务：把 Chrome DevTools 接入Agent，让Agent自己去看开发的结果，然后反馈、修复。

### 思维差异
当Agent反馈不符合要求的结果时：

- **传统思维**：立即告诉它哪里错了，让它修复
- **Harness思维**：思考"我应该构造一个什么环境，让它自己知道做得有问题"

## 第二点：面向智能体建设文档系统

### 核心目标
让智能体在面对任务/问题时，可以：
1. 首先知道去哪开始了解任务
2. 通过渐进式披露的方式，逐渐输出方案

### 建设要点

- **AGENTS.md**：作为整个仓库的目录，说明代码和文档的位置，作为路由指导Agent去对应位置继续阅读
- **持续建设**：整个仓库需要建设文档，并持续检查文档正确性，补充、修复文档

### 本质
相当于给Agent建设一个**操作手册**

### 重要原则：文档要拆分，不要大而全
- 大而全会造成Agent一次阅读过多内容，没有重点
- 小而分散，并辅以路由，可以让Agent快速找到自己需要的内容

## 第三点：持续维护规范

### 核心逻辑
Agent会阅读已有代码，按照已有范式新增代码。

### 关键点
- 一定要持续完善规范，约束Agent的行为
- 否则仓库代码会递归式的越来越乱

### 熵增风险
如果不做持续的规范维护：
- 代码会不断根据已有代码模式进行**复制、扩张**
- 持续**熵增**

---

## 第四点：先通过后修复的工程流程

### 传统模式
由于开发/修复代码的成本较高，倾向于在合入之前**反复确认**没有错误。

### Agent时代的模式
- 开发/修复代码的成本较低
- 写代码、合代码的吞吐比较高
- 长时间阻塞会降低迭代效率

### 转变
**先合入，再快速修复**，尽量减少阻塞点。
