# Agent 普及困难的核心矛盾

## 事件背景

虽然我们构建了一堆 skill、agent 配置，但没人真的用。

这不是工具的问题，是期望错配的问题。

## 核心矛盾

**用户想要的是：说问题 → 得答案**
**Agent 要求的是：说问题 → 理解 agent 逻辑 → 调整提问方式 → 多轮交互 → 才能得到答案**

这两者之间存在根本性的摩擦。

## 三个致命原因

### 1. 学习成本与用户心态相悖

Agent 虽然看着强大，但用它需要理解 agent 的运行方式：上下文窗口、tool use、multi-step reasoning……

而大多数人的心态是：**只想完成任务，不想学习工具本身**。

这不是懒，是理性选择。如果完成任务之前还要先上一堂课，这个工具就输了。

### 2. Skill 变成了"需要维护的负担"

我们构建了很多 skill，想着"这个能力以后会有用"。

但现实是：
- Skill 需要按它的逻辑去写和维护
- 用 skill 之前需要理解它能做什么、不能做什么
- 当 skill 行为不符合预期时，需要调试

**一个需要你持续投入精力去维护的工具，不是工具，是第二个工作。**

### 3. 多轮沟通本身就是成本

人想要的对话是：

> "帮我把这段代码重构一下"
> "好，改完了"

而不是：

> "帮我把这段代码重构一下"
> "你是想只改函数命名，还是包括变量名？"
> "包括"
> "你希望保持原有逻辑，还是可以接受一定程度的优化？"
> "可以优化"
> "那我先改函数命名，改完你确认一下？"

**每多一轮沟通，就多一分认知负荷。**

## 深层问题：中间层必须足够透明

Agent 作为人和目标之间的中间层，它存在的价值是：

```
人 → Agent → 目标
```

但如果这个中间层需要人去伺候，那它就不是在帮忙，是在添乱。

**中间层要真正有用，必须做到：清晰到不需要人干预。**

要么你足够简单，简单到不需要学习；要么你足够可靠，可靠到不需要监督。二者至少占一个。

## 结论

Agent 工具本身的成熟度不是问题，**问题是它对用户的期望太高了**。

当用户的心态是"我说话，你干活"，而 agent 的实际行为是"你说清楚，我再干"，这个鸿沟就会让工具被搁置。

**真正被广泛使用的 agent，不是功能最强的，而是门槛最低的。**
