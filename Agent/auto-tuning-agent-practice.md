# 推荐排序策略服务 Agent 实践分享：自动调参

## 前言

说到 Agent 落地，我认为首先不是问"模型强不强"，而是先问两个问题：

- **我们要让 Agent 干什么？**
- **Agent 能干什么？**

前者需要从需求出发分析，后者需要在不断用 Agent 解决问题过程中完善能力。

---

## 一、需求背景

推荐排序策略服务团队负责推荐系统稳定性、架构迭代和指标提升等方方面面。推荐排序策略服务的特点是**迭代人数多、迭代强度高、服务规模大**。

在推荐排序策略服务中，**自动调参**是一个重要的探索式任务场景：无明确触发，需要 Agent 主动寻找系统中的优化空间。

---

## 二、背景与痛点

推荐排序策略服务有 **35715 个 AB 参数**、**81 个 热配置中心**、**378 个 F1 公式**。调参是推荐排序策略服务非常重要的一部分，但面临以下挑战：

- **搜索空间大**：需了解大量参数含义才能做决策
- **任务差异大**：经验不易在任务间迁移
- **严重依赖专家经验**：先调哪些参数、现象对应什么方向、换了任务怎么迁移

---

## 三、传统方法的局限

### 贝叶斯优化（BO）

贝叶斯优化本质是在预先定义好的搜索空间里找更优解。但在工业复杂场景里有两个主要难点：

1. **前期仍需大量人工工作**：决定哪些参数值得调、范围怎么设、不同参数间的约束
2. **BO 更擅长处理"已经参数化好的问题"**：当需要动态生成新公式、新规则组合、甚至代码逻辑时，优化对象不再是"调参数"而是"改结构"，BO 就不擅长了

例如：

- 如果调高 w_1，指标变好了，BO 可以继续沿着这个方向搜索
- 但如果把公式从"加法"改成"乘法"之后指标变好了，这种"结构变了"的信息，很难被传统 BO 直接利用起来

---

## 四、LLM/Agent 做调参的优势

| 维度 | 传统 BO | LLM/Agent |
|------|---------|-----------|
| 前置经验 | 需人工定义搜索空间 | 结合预训练知识 + 代码/配置/日志/实验记录，自动分析哪些参数值得调 |
| 搜索方式 | 在固定空间里数值搜索 | 根据现象生成新候选结构（公式、规则组合、代码逻辑） |
| 决策方式 | 根据历史分数推算 | 读取历史实验记录、训练日志，先解释现象，再提出候选 |

**Agent 相对 LLM 的进化**：LLM 解决"理解和生成候选方案"，Agent 进一步把理解、分析、搜索、工具调用、实验执行串成闭环。

---

## 五、推荐排序策略服务的尝试

通过一个实验走通了流程：

```
Agent 修改 热配置中心
    │
    ▼（等待 2 小时获取 AB 指标）
    │
调用 ClickHouse MCP 查询实验指标
    │
    ▼
LLM 决策：参数上调还是下调，修改 热配置中心
    │
    ▼
循环，直到符合终止条件
```

这个实验打通了 Agent 自动调参的流程。

**流程说明**：

1. Agent 根据当前参数配置修改 热配置中心
2. 等待 2 小时获取 AB 指标
3. 通过 ClickHouse MCP 查询实验指标
4. LLM 根据数据表现决定参数是上调还是下调，并修改 热配置中心
5. 进行下一轮循环，直到参数符合终止条件

## 六、遇到的问题及待解决项

| 问题 | 说明 |

| 如何找最优参数 | 代码仓库大、参数多，如何通过离在线联动分析做功位置 |

| 各平台如何面向 Agent 重新设计参数的上下线流程 | 流程适配 |

## 七、总结与后续方向

当前已打通 ClickHouse、热配置中心 链路，Agent 自动调参流程已跑通。后续继续推进：

- 联动更多平台
- 利用 Agent 提升迭代效率
- 探索 MCP/CLI 接入方式
- 设计面向 Agent 的参数上下线流程

Agent 在自动调参领域潜力巨大，将持续探索和完善。
