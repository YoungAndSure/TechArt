# 从一次Coredump看工程职责边界

## 事件背景

某服务中，算法团队负责策略开发，架构团队负责稳定性、性能和可维护性。代码流程是：算法提CR → 架构review → 合入团队分支跑测试 → 合入master → 灰度观察 → 正式发布。

某次架构review通过后，合入团队分支时发生了coredump。领导的反馈是："这个代码没加开关，review时还要多把关下。"

我理解领导的回复意思是，需要架构reviewer加强review以避免此类问题发生。

我认为这个归因值得商榷。

## 职责边界在哪里

算法写策略代码，第一责任人应该是代码作者。架构review的职责通常包括：

- 是否破坏公共约束
- 是否有明显稳定性/性能/可维护性风险
- 是否符合模块规范
- 建设机制和工具，降低整体出错率

架构review不是替作者做"逐行验收"，更不是替作者对业务逻辑和实现细节兜底。

该服务已有一整套链路：dryrun测试 → 团队分支流程测试 → master → 灰度 → 正式。这套机制本身说明：质量不是靠某一个reviewer人肉兜住，而是靠多层机制拦截。

团队分支测出coredump，恰恰说明机制在发挥作用。

## Review本来就不是最优手段

有150+算法同学，一天几十个CR，峰值54个，架构不可能对每个需求的上下文、每段实现细节、每个潜在的内存/边界/空指针/并发问题都深入理解。这不是态度问题，是信息论上做不到的事：

- reviewer不掌握完整需求上下文
- reviewer不是代码作者，脑内没有实现路径
- reviewer处理量太大，不可能深度阅读全部变更
- coredump类问题更适合靠测试、静态分析、规则扫描来发现

"靠更仔细的人肉review来解决coredump"是低扩展性方案。短期看像在负责，长期看是在拿人的注意力填系统漏洞。

## 归因的分层思考

"reviewer没看出来"和"测试阶段发现coredump"之间，没有必然的因果关系。

更合理的归因应该是：

1. **作者责任**：该加开关没加，这是代码作者的直接责任
2. **review责任**：如果这类变更通常都要求加开关，reviewer可以补充检查
3. **机制责任**：把"涉及策略生效路径必须加开关"沉淀成模板、lint规则、agent规则、CR checklist

领导说的"这种代码没加开关，review时还要多把关下哈"，可以理解为两层意思：

- **经验反馈**：这类容易出风险的点，以后review时重点关注。合理。
- **责任倒灌**：因为你没看出来，所以主要是你的问题。不合理。

## 架构应该做什么

架构的核心职责，不是替所有作者更仔细地看代码；而是建设一套机制，让即使作者会犯错、review不可能完美，问题也能尽量早暴露、少暴露、别流到线上。

每次coredump后应该反推：

- 这是哪类模式？
- dryrun为什么没拦住？
- 团队分支测试拦住了，但能不能再前置？
- agent review能不能补一条规则？
- 是否要加静态检查、单测模板、变更准入项？

把问题当成"规则建设素材"，而不是"谁没认真"的证据。

## 结论

- 作者第一责任
- 机制第二责任
- review是辅助责任，不是兜底责任

让系统进化，而不是压榨人的注意力。
